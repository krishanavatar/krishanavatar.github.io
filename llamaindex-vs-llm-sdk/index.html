LLM Engineering — Framework Decisions

LlamaIndex or the SDK: When to Use Each

The framework does real work. Whether that work matches the problem you have — and whether you will carry its overhead on every request regardless — is the question most teams skip before they start building.

krishanavatar.github.io / July 2026 / 10 min read
THESIS

LlamaIndex is not a smarter way to call an LLM. It is an orchestration framework for data pipelines that feed LLMs. If your problem is pipeline complexity, it pays off. If your problem is retrieval quality on a simple corpus, it introduces costs you do not need.

The Decision in Three Rules
  • 1
    One data source, one retrieval step — use the SDK directly. A complete RAG pipeline fits in 100 lines of Python. You control everything. The framework saves nothing here.
  • 2
    Multiple data sources, routing across indexes, or agents — LlamaIndex pays off. Its connector library and query routing infrastructure save weeks on problems that would otherwise require custom scaffolding.
  • 3
    You do not have to choose completely. LlamaIndex's data and indexing layers compose independently. Use the connectors for ingestion, then call the Anthropic SDK for generation. The framework owns the pipeline; you own the prompt.
01 — The Setup

The Wrong Question

Most engineers approach this as a tool preference question: LlamaIndex or the raw API? That frames it wrong. The real question is: how complex is the data layer of the application I am building?

Simple data layer — one corpus, one embedding model, one retrieval step, one prompt — and the API client is all you need. A production-grade RAG pipeline for a single document store fits in 100–120 lines of Python using the Anthropic or OpenAI SDK directly. It has no hidden behavior, no framework overhead, and every step is something you wrote and can instrument.

Complex data layer — multiple source systems, document stores that update incrementally from several connectors, different query types that should route to different indexes, agents that invoke retrieval as one tool among several — and LlamaIndex is solving a problem that would otherwise require significant custom scaffolding. That is where the framework investment pays.

The confusion arises partly because LlamaIndex markets itself as a RAG library. It is also that, and it is a great deal more. Using it purely for a single-source RAG pipeline is paying the overhead of an orchestration framework to solve a retrieval problem that does not require orchestration.

02 — The Framework

What LlamaIndex Actually Is

LlamaIndex has four major layers. Understanding what each layer does — and which layer you actually need — is the starting point for any framework decision.

Data connectors. LlamaIndex ships 150+ readers: Notion, Confluence, S3, PostgreSQL, GitHub, web crawlers, PDF parsers, Slack, Google Drive. Each handles its own auth flow, pagination, rate limiting, and document schema normalization. The output is a uniform list of Document objects regardless of source. Building the same breadth SDK-direct means implementing each loader from scratch.

Indexing layer. VectorStoreIndex for semantic similarity, SummaryIndex for long-document compression, KeywordTableIndex for exact-term retrieval, KnowledgeGraphIndex for entity-relationship queries. Each index type has different retrieval characteristics and different costs. LlamaIndex provides a consistent interface across all of them.

Query engines and routers. A QueryEngine wraps an index and handles the retrieval-to-generation pipeline. A RouterQueryEngine classifies the incoming query and dispatches it to the appropriate index. This routing layer is what separates LlamaIndex from a simple vector search wrapper. Query routing across heterogeneous indexes is the kind of infrastructure that takes weeks to build correctly without a framework.

Agents. ReActAgent and FunctionCallingAgent expose query engines as callable tools in an agent loop. The agent decides when to invoke retrieval, what query to issue, and how to synthesize results across multiple tool calls. This is where LlamaIndex converges with the broader question of agentic architectures.

The layers compose independently. I can use the data connectors to ingest documents, build a vector index, retrieve nodes, and then pass those nodes to the Anthropic SDK for generation without touching LlamaIndex's query engine or response synthesizer. This composability is useful and frequently overlooked.

Layer 1

Data Connectors

150+ readers for Notion, Confluence, S3, databases, GitHub, web. Handles auth, pagination, incremental updates.

Layer 2

Index Types

Vector, Summary, Keyword, Knowledge Graph. Each retrieves differently. One interface across all.

Layer 3

Query Routing

RouterQueryEngine dispatches across multiple indexes. Intent classification built in.

03 — The Alternative

What SDK-Direct Means

SDK-direct means using the Anthropic SDK, OpenAI SDK, or equivalent to call the model API without an orchestration layer. The SDK handles authentication, request construction, retries on transient errors, streaming, and token counting. Everything else — chunking, embedding, retrieval, prompt construction, context assembly — is code you write.

That sounds like more work. For a simple pipeline, it is not. The pipeline is explicit. A complete, production-ready single-source RAG implementation in the Anthropic SDK is straightforward to write, instrument, and debug.

Python — complete SDK-direct RAG pipeline, single source
import anthropic
import numpy as np
from typing import List

client = anthropic.Anthropic()

def cosine_sim(a: np.ndarray, b: np.ndarray) -> float:
    return float(np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b) + 1e-9))

def embed(texts: List[str], model: str = "text-embedding-3-small") -> List[np.ndarray]:
    # swap for your embedding provider
    import openai
    resp = openai.embeddings.create(input=texts, model=model)
    return [np.array(r.embedding) for r in resp.data]

def retrieve(query: str, chunks: List[str], chunk_embeddings: List[np.ndarray], k: int = 5) -> List[str]:
    [q_emb] = embed([query])
    scores = [cosine_sim(q_emb, c_emb) for c_emb in chunk_embeddings]
    top_k = sorted(range(len(scores)), key=lambda i: scores[i], reverse=True)[:k]
    return [chunks[i] for i in top_k]

def answer(query: str, chunks: List[str], chunk_embeddings: List[np.ndarray]) -> str:
    context_chunks = retrieve(query, chunks, chunk_embeddings)
    context = "\n\n---\n\n".join(context_chunks)

    response = client.messages.create(
        model="claude-sonnet-4-5",
        max_tokens=1024,
        system="Answer using only the provided context. If the answer is not in the context, say so.",
        messages=[{"role": "user", "content": f"Context:\n{context}\n\nQuestion: {query}"}]
    )
    return response.content[0].text

# Usage: load chunks at startup, pass pre-embedded list on each call
# Every step is visible, loggable, and replaceable without framework negotiation

This is not a toy implementation. It is the actual pipeline. Swap the embedding call for your vector store's native search, add a reranker pass between retrieve and answer, and this scales to millions of documents. The code is yours. When it fails, you know exactly where to look.

04 — The Trade-Off

The Abstraction Cost

Frameworks impose an abstraction tax. For LlamaIndex specifically, the tax has three components that compound in production.

Hidden behavior. LlamaIndex constructs prompts internally. The default prompts for query engines, response synthesizers, and agents are documented but opaque at the call site. When a query engine produces an unexpected answer — truncated, mis-attributed, wrong tone — finding where the response synthesizer constructed its prompt takes investigation. SDK-direct pipelines have no such layer. The prompt you write is the prompt that runs.

Version churn. LlamaIndex has undergone two major API surface changes in three years: from gpt_index to llama_index to the llama_index.core split across sub-packages. Class names, import paths, and parameter signatures changed between minor versions within each era. If your production system pins an older version, you fall behind on bug fixes. If you upgrade, you may break working code. The SDK client has a much more stable interface because it maps closely to the underlying HTTP API.

Debugging distance. When a query engine returns a wrong answer, the failure could be in the node retriever, the node postprocessors, the response synthesizer, the underlying LLM call, or the prompt template. The framework owns the call stack. Adding observability requires callback hooks or an instrumentation layer. SDK-direct pipelines have no indirection. I add a print or a structured log exactly where I need it.

These costs are real but not disqualifying. They are worth paying when the framework is solving a problem that would otherwise require proportionally more custom work. The question is always: what problem is the framework solving, and is that the problem I have?

The Debugging Test

Before adopting a framework, ask: when this pipeline returns the wrong answer in production at 2am, how quickly can I find the fault? SDK-direct: add a log line, redeploy, done. LlamaIndex: configure a callback handler, trace through the node pipeline, inspect the synthesized prompt. Both are solvable. One is faster under pressure.

05 — Where the Framework Pays

Where LlamaIndex Pays Off

There are specific scenarios where LlamaIndex's cost is dwarfed by the problem it solves.

Multi-source data ingestion. Notion for product specs, Confluence for engineering documentation, S3 for PDF contracts, a PostgreSQL table for structured records. Each source needs its own auth flow, pagination strategy, incremental update mechanism, and document normalization step. LlamaIndex's connector library covers all of these with maintained, tested implementations. Building the same capability SDK-direct means writing and maintaining four separate loaders. I have done this. On a project at Infrrd that needed five source systems, using the LlamaIndex connector ecosystem saved approximately three weeks of integration work.

Routing across multiple indexes. A support system where policy questions should retrieve from the policy document index, procedural questions from the FAQ index, and pricing questions from a structured data table. Routing by query intent requires intent classification, index dispatch, and result synthesis. LlamaIndex's RouterQueryEngine with an LLMSingleSelector handles this in twelve lines. SDK-direct requires building the classification and dispatch layer from scratch, which is not complex but is work that the framework has already done correctly.

Python — multi-source ingestion with LlamaIndex connectors
from llama_index.core import VectorStoreIndex
from llama_index.readers.notion import NotionPageReader
from llama_index.readers.confluence import ConfluenceReader
from llama_index.readers.s3 import S3Reader

# Three sources, one index — this is where LlamaIndex earns its overhead
notion_docs = NotionPageReader(integration_token=NOTION_TOKEN).load_data(
    page_ids=["page-id-1", "page-id-2"]
)
confluence_docs = ConfluenceReader(base_url=CONFLUENCE_URL, token=CONF_TOKEN).load_data(
    space_key="ENG"
)
s3_docs = S3Reader(bucket="my-contracts", prefix="active/").load_data()

all_docs = notion_docs + confluence_docs + s3_docs
index = VectorStoreIndex.from_documents(all_docs)

# Query across all three sources transparently
engine = index.as_query_engine()
response = engine.query("What is the SLA for enterprise contracts?")
Python — router query engine across heterogeneous indexes
from llama_index.core.query_engine import RouterQueryEngine
from llama_index.core.selectors import LLMSingleSelector
from llama_index.core.tools import QueryEngineTool

policy_tool = QueryEngineTool.from_defaults(
    query_engine=policy_index.as_query_engine(),
    description="Insurance policy terms, coverage limits, exclusion clauses"
)
faq_tool = QueryEngineTool.from_defaults(
    query_engine=faq_index.as_query_engine(),
    description="General product questions, onboarding, account management"
)

# LLMSingleSelector classifies the query; RouterQueryEngine dispatches it
router = RouterQueryEngine(
    selector=LLMSingleSelector.from_defaults(),
    query_engine_tools=[policy_tool, faq_tool]
)

# No manual intent classification — the router decides which index to hit
response = router.query("Does flood damage count as an exclusion under my policy?")

Agents with retrieval as a tool. When the application needs an agent that can both retrieve documents and call external APIs — a support agent that looks up policy language, checks a customer's account status, and drafts a response — LlamaIndex's agent framework composes retrieval tools with API tools cleanly. SDK-direct tool-use with the Anthropic SDK is also viable here, but LlamaIndex's integration between query engines and the agent tool abstraction reduces the wiring work.

Evaluation and observability integrations. LlamaIndex ships built-in evaluators for faithfulness, relevance, and correctness, along with integrations to tracing backends like Arize Phoenix, LangFuse, and Weights & Biases. If your team wants structured retrieval evals with minimal setup, this is faster than building an eval harness SDK-direct.

06 — Where the SDK Wins

Where SDK-Direct Is the Right Call

SDK-direct is the correct choice in four specific situations, and in each case the reason is different.

Single-source, single-step pipelines. One document store, one embedding model, one retrieval step. The framework overhead does not simplify a pipeline that is already simple. A LlamaIndex VectorStoreIndex wrapping a Pinecone client has more moving parts than calling the Pinecone client directly, with no additional capability for this use case. SDK-direct is shorter, faster, and fully transparent.

Latency is a constraint. LlamaIndex framework overhead — object initialization, node processing pipeline, response synthesizer dispatch — adds measurable latency compared to an equivalent SDK-direct call. I measured 160–220ms of pure framework overhead on a simple LlamaIndex VectorStoreIndex query engine versus the same retrieval and generation steps written SDK-direct against the same Pinecone index. On a conversational product where first-token latency is felt by the user, this overhead is not free.

Custom pipeline logic is the product. If the chunking strategy, reranking algorithm, or context assembly method is proprietary — the differentiated IP of the application — it is easier to own that code than to adapt it to the framework's node and postprocessor abstractions. LlamaIndex's NodeParser and BaseNodePostprocessor interfaces work, but imposing domain-specific logic through framework extension points adds friction that is not present in code you write directly.

You need to debug under production pressure. The SDK call stack is flat. You call an API, you get a response, everything in between is your code. LlamaIndex's call stack has multiple abstraction layers between your query and the model call. When something fails on a Friday night, the debugging path matters.

Framework Cost

160–220ms overhead per request

Measured on a simple VectorStoreIndex query engine vs equivalent SDK-direct pipeline. Grows with more postprocessor steps.

SDK Advantage

Flat call stack, zero framework magic

Every line between the user query and the model call is yours. Log any step, replace any step, debug any step without callback hooks.

The small corpus trap

The most common LlamaIndex over-engineering I see: teams build a full framework pipeline for an internal knowledge base of 50–200 documents. For a corpus that small, pass the entire thing in a long-context window and skip retrieval entirely. Full-context on 200 pages fits in Claude's context window. RAG adds retrieval risk to a corpus small enough to avoid it. LlamaIndex adds framework overhead on top of that. The correct tool for 200 documents is a 200k-token context and one API call.

07 — The Better Pattern

The Hybrid Pattern

The most underused option: use LlamaIndex for data ingestion and indexing, then use the SDK for generation. You get the connector ecosystem without ceding control over the prompt.

LlamaIndex ingests your documents, handles connector authentication, normalizes document schemas, and builds the vector index. On a query, you call retriever.retrieve(query) to get the top-k nodes. You take those nodes, construct the prompt exactly as you want it, and call the Anthropic SDK for the generation step. LlamaIndex never touches the model call.

Python — LlamaIndex for retrieval, Anthropic SDK for generation
import anthropic
from llama_index.core import VectorStoreIndex
from llama_index.readers.notion import NotionPageReader

# LlamaIndex handles ingestion and indexing
docs = NotionPageReader(integration_token=NOTION_TOKEN).load_data(page_ids=PAGE_IDS)
index = VectorStoreIndex.from_documents(docs)
retriever = index.as_retriever(similarity_top_k=5)

# You handle generation — full prompt control
anthropic_client = anthropic.Anthropic()

def answer(query: str) -> str:
    nodes = retriever.retrieve(query)
    context = "\n\n".join(node.text for node in nodes)

    # Your prompt, exactly as you want it — no framework synthesizer
    system = (
        "You are a technical assistant for the engineering team. "
        "Answer only from the provided context. "
        "If the answer is not in the context, say 'I do not have that information.'"
    )
    user = f"Context:\n{context}\n\nQuestion: {query}"

    response = anthropic_client.messages.create(
        model="claude-sonnet-4-5",
        max_tokens=1024,
        system=system,
        messages=[{"role": "user", "content": user}]
    )
    return response.content[0].text

# LlamaIndex owns the data layer. Anthropic SDK owns the model call.
# Neither owns the other's domain.

This is not a workaround. LlamaIndex explicitly supports using the retriever as a standalone component. The as_retriever() method returns a retriever that is independent of the query engine and response synthesizer. The framework's author documented this path because it is a legitimate usage pattern, not a misuse of the library.

The hybrid pattern is my default recommendation when a team needs multi-source ingestion but wants to maintain prompt control. It gives you LlamaIndex where LlamaIndex is valuable — the data pipeline — and the SDK where the SDK is valuable — the model call.

08 — The Framework

Decision Framework

The decision is driven by data pipeline complexity, not preference. These are the conditions that should move the needle.

Condition LlamaIndex SDK-Direct Why
Single data source, single retrieval step No Yes Framework overhead adds no capability. SDK-direct is shorter and debuggable.
Multiple data sources (3+) with different connectors Yes No Connector library saves weeks of loader implementation. This is LlamaIndex's core strength.
Routing queries across heterogeneous indexes Yes No RouterQueryEngine with LLMSingleSelector is 12 lines. SDK-direct requires building classification and dispatch from scratch.
Agents with retrieval as one of many tools Yes Maybe LlamaIndex composes retrieval into agent tool abstractions cleanly. Anthropic SDK tool-use also works but requires more wiring.
Latency-sensitive path (<1s first-token target) No Yes LlamaIndex adds 160–220ms overhead per request. On latency-sensitive paths, this is measurable and felt.
Custom chunking or embedding logic as core IP No Yes Extending framework node parsers adds friction. Owning the chunking code is cleaner when the logic is proprietary.
Corpus under 200 documents No No Pass everything in a long-context window and skip retrieval. RAG adds risk to a corpus small enough to avoid it entirely.
Need multi-source ingestion but want prompt control Partial Partial Use the hybrid pattern: LlamaIndex for ingestion and retrieval, Anthropic SDK for the generation call.
Built-in retrieval evals and observability Yes No LlamaIndex ships faithfulness and relevance evaluators and integrates with tracing backends. SDK-direct requires building or wiring your own.
Start with the SDK, migrate when the pipeline grows

The lowest-risk path: start with the Anthropic SDK for your first production RAG system. When you add a second data source with a different connector, evaluate LlamaIndex's reader for that connector. When you need to route queries across two indexes, evaluate RouterQueryEngine. Adopt framework components incrementally as they solve problems you actually have, not problems you anticipate having.

09 — Takeaway

The Frame to Leave On

LlamaIndex is an excellent framework for the problems it is designed to solve. Multi-source ingestion, query routing across heterogeneous indexes, and agents with retrieval tools are genuinely hard to build well without the infrastructure it provides. Reaching for it on those problems is the right call.

For a single-source retrieval pipeline, the Anthropic SDK with your own chunking and retrieval code is shorter, faster, and easier to debug. The framework does not make a simple pipeline better. It makes it more opaque. That trade is only worth making when the framework is solving complexity that would otherwise require significant custom work.

The question to ask before choosing is not which tool is more capable. It is: what is the irreducible complexity of this pipeline, and does the framework address it?
