SOLID Principles in Production: Why I Use Them, and What They Cost

Five rules from 2000 that predate microservices, most modern frameworks, and every LLM API you're calling today. I still reach for them weekly. Here is the actual case for using SOLID, the tax nobody mentions in the talks, and how I decide when a principle is worth invoking.

krishanavatar.github.io / July 2026 / 9 min read
THESIS

SOLID is not a checklist to satisfy on every class you write. It is a set of forces you invoke when change, not readability, is the actual problem you're solving.

The Five Letters, Fast
  • S
    Single Responsibility — a class has one reason to change. Mixing OCR extraction, confidence scoring, and review-queue routing in one class means a scoring tweak risks breaking extraction that already works.
  • O
    Open/Closed — extend behavior by adding code, not editing a central switch. A growing if/elif on a type string is the smell.
  • L
    Liskov Substitution — a subtype has to honor every guarantee its base type made. A subclass that raises where the parent promised success breaks every caller written against the base contract.
  • I
    Interface Segregation — don't force a class to implement methods it can't support. Small, role-shaped interfaces beat one interface with five stub methods.
  • D
    Dependency Inversion — high-level logic depends on an abstraction, not a concrete class. The concrete choice gets wired in exactly once, at the edge of the program.
01 — Context

The Principles Everyone Cites, Few Apply Well

SOLID gets cited in interviews more than it gets practiced in production. Most engineers can recite the five letters. Fewer can tell you which one they'd invoke on the file open in front of them right now, and why it applies there and not on the file next to it.

Robert Martin named these principles in the early 2000s, pulling together ideas that had been circulating in the object-oriented design community since the 1980s. They predate dependency injection frameworks, they predate microservices, and they certainly predate calling an LLM API from inside a request handler. The fact that they still hold up says something. So does the fact that most teams apply them inconsistently: rigorously in the auth module, not at all in the script that reconciles Friday's billing export.

I built a small companion repository with five runnable Python examples, one per letter, each with a before/after: github.com/krishanavatar/soild-principle. I'll pull two of those scenarios into this post because a principle argued in the abstract is easy to nod along to and hard to actually use.

02 — The Case For

Why I Still Use It in Production

The honest reason I reach for SOLID isn't elegance. It's that I've been paged for the alternative.

On our document intelligence pipeline at Infrrd, every page gets run through an OCREngine interface before extraction. When a specific document type needed a different OCR provider to hit acceptable confidence scores, the fix was a new class implementing that same interface, routed in for that document type. The pipeline that calls it never changed. The extractors for every other document type never changed. Nothing that already worked in production needed to be retested, because nothing that already worked was touched. That is the Open/Closed principle paying for itself on real documents, not in theory.

Single Responsibility earns its keep at onboarding time more than at write time. A new engineer reading a ConfidenceScorer that only scores extracted fields can trust it does nothing else. They don't have to trace through OCR calls or review-queue routing to convince themselves a scoring-threshold change is safe. That confidence compounds across a team doing code review on each other's pull requests in the same pipeline.

Dependency Inversion is the one that saves me in tests, every day, not just on the day an OCR provider changes. Our extraction service is built against that same OCREngine protocol, so its test suite takes a fake OCR engine and asserts on what got returned, no real OCR call, no network round trip, no flaky CI run because a vendor's sandbox is down. I don't do this because a textbook told me to. I do it because I've maintained pipelines where a test suite depended on a live OCR endpoint, and I know what a Tuesday morning looks like when that endpoint is down.

03 — The Cost

The Cost Nobody Puts in the Talk

Every SOLID conference talk shows the payoff. Almost none shows the invoice.

The first cost is indirection. An interface, a factory, and three implementations for an OCR engine that will only ever run one vendor's library is harder to read than the direct version, not easier. I've opened pipelines where finding the actual OCR call meant walking through an engine interface, a factory, and a dependency-injection container registration, four files for one call to a library nobody ever swapped out. That is Dependency Inversion applied to a dependency that was never going anywhere.

The second cost is premature generalization disguised as Open/Closed. I've seen a DocumentClassifier built with a pluggable strategy pattern for five model backends when the pipeline only ever shipped one YOLO-based classifier. The plugin architecture existed to serve backends nobody had requested. Long after, nobody had added a second backend, and the abstraction was pure overhead on every change to the one classifier actually in use.

The third cost is speed of shipping the first version. If you are validating whether a feature has any users at all, wiring up five interfaces before you've proven the feature is worth building is optimizing for a change that a failed experiment will never need.

The tell that you've over-applied it

If you cannot explain, out loud, what concrete second implementation an interface exists for, you have built abstraction for its own sake. "We might need it later" is not a second implementation. A second implementation is a second implementation.

04 — Balance Sheet

Pros and Cons, Without the Sales Pitch

Laid out side by side, the trade is legible. It's a trade, not a free upgrade, which is exactly why it needs judgment instead of a rule that always applies.

Principle What you gain What it costs
Single Responsibility Smaller units, easier review, isolated test failures More files; a simple task can span three of them
Open/Closed New behavior added without touching, or breaking, old behavior An interface and implementation for variants that may never arrive
Liskov Substitution Callers can trust the base contract; no surprise exceptions Forces you to model a "different" case as a new type, more upfront design
Interface Segregation No stub methods, no lying about capability More interfaces to track; easy to over-split into single-method traits
Dependency Inversion Swappable dependencies, fast unit tests with fakes A layer of indirection between the caller and the thing that does the work
05 — Applicability

Where It's Worth It vs Where It's Overkill

The variable that matters is not the size of the codebase. It's whether the part you're touching is expected to change, and whether more than one person depends on the contract staying stable.

Worth it

OCR engines, document-type extractors, review-queue notifiers

Concrete history of swapping providers per document type, or a near-certain roadmap item to add one. Multiple call sites depend on the same contract staying stable.

Overkill

A one-off backfill script that re-scores last quarter's documents

Single author, single run, or a lifespan measured in weeks. The abstraction outlives the code it was meant to protect.

Worth it

Anything with more than two engineers reading and modifying it regularly

Single Responsibility reduces the blast radius of one person's change on another person's area. That coordination cost is real and SOLID is buying it down.

Overkill

Per-tenant confidence thresholds that change monthly by design

If the "variation" is actually data (thresholds, feature flags, per-document-type limits), model it as data. A polymorphic class hierarchy for something that's really a lookup table adds ceremony with no benefit.

06 — Worked Example

One Principle, One Real Refactor

Dependency Inversion is the principle that shows its value most clearly in a diff, so here's the same OCREngine boundary from earlier, simplified down to the shape that matters. The starting point: a DocumentProcessor that constructs its own OCR engine.

Python — before: welded to a concrete OCR engine
class DocumentProcessor:
    def __init__(self):
        # the high-level class now owns a low-level detail
        self._ocr = TesseractOCR()

    def process(self, document):
        text = self._ocr.read(document)
        return score_confidence(text)

Testing this means either mocking TesseractOCR internals or running a real OCR call in your test suite. Routing one document type to a different OCR provider means editing DocumentProcessor itself. The fix inverts the dependency: DocumentProcessor depends on the OCREngine protocol, and the concrete engine is injected from outside.

Python — after: depends on OCREngine, injected at the edge
class OCREngine(Protocol):
    def read(self, document) -> str: ...

class DocumentProcessor:
    def __init__(self, ocr: OCREngine):
        self._ocr = ocr   # chosen by the caller, not by this class

    def process(self, document):
        text = self._ocr.read(document)
        return score_confidence(text)

# composition root: the ONE place that knows the concrete engine
processor = DocumentProcessor(TesseractOCR())

# routing a different document type to a different provider needs
# no change to DocumentProcessor itself
processor = DocumentProcessor(AzureFormRecognizer())

# tests: no OCR call, no network, just a fake that returns fixed text
processor = DocumentProcessor(FakeOCREngine())

Four lines changed the shape of the class. The gain is not aesthetic. It's that DocumentProcessor's test suite now runs in milliseconds with zero external calls, and routing a new document type to a different OCR provider later touches zero lines inside DocumentProcessor.

07 — Decision Rule

The Heuristic I Actually Use

I don't open a new file asking "which SOLID principles should I apply." I write the direct version first. I reach for a principle only once I can point to the specific pain it removes.

  • Apply Single Responsibility when a class's test setup requires mocking more than one unrelated thing (an OCR engine and a notification service for what's supposed to be a confidence-scoring function).
  • Apply Open/Closed once you're adding the second document type or the second OCR provider, not before the first one exists. One implementation doesn't need an interface.
  • Apply Liskov Substitution as a check, not a pattern to add code for: before you subclass, ask whether every existing caller of the base type still gets what it was promised.
  • Apply Interface Segregation the moment you write raise NotImplementedError in a method your class was forced to implement, like a read-only document archive forced to implement a write method it will never support. That line is the signal, not a red flag to just accept.
  • Apply Dependency Inversion at any boundary that talks to something outside your process: an OCR engine, a database, an LLM API, a queue. Inside a single pure function, it's rarely worth it.
Retrofit over predict

Every case above is diagnosed from evidence already in front of you, a real second implementation, a real mocking headache, a real stub method. None of them require predicting the future. That's the difference between applying SOLID and performing it.

08 — The Team Dimension

The Team Dimension

Everything above is framed at the level of one engineer and one file. The variable that shifts the math most in practice is team size, and specifically, how many people touch the same module without talking to each other first.

On a team of two who sit next to each other, a shared mental model substitutes for a lot of structural enforcement. On a team of eight spread across two time zones, the interface boundary is doing communication work that a hallway conversation would otherwise do. The same class, written the same way, is under-engineered on the first team and correctly engineered on the second.

This is why I don't think SOLID is a property of code in isolation. It's a property of code relative to how many people depend on its contract staying still while they're not looking.

09 — Takeaway

The Frame to Leave On

SOLID is a set of five answers to five specific kinds of pain: a class that keeps changing for unrelated reasons, a switch statement that keeps growing, a subclass that breaks its caller's assumptions, an interface that forces dishonest stub methods, and a high-level class welded to a low-level detail.

Use a principle when you can name the pain it removes on the code in front of you. Skip it when the only justification is a future you're guessing at. The five letters are a diagnosis, not a tax you owe on every file.

The runnable examples are in the repo if you want to see all five with real before-and-afters: github.com/krishanavatar/soild-principle.
