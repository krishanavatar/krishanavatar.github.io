Production Engineering — Retrieval Systems

Why RAG Pipelines Fail in Production

The gap between a working demo and a reliable production system is wider than most teams estimate. Here is where the breaks actually happen, and what to do about each one.

krishanavatar.github.io / July 2026 / 10 min read
THESIS

Most RAG failures are not retrieval failures. They are system design failures that happen to surface during retrieval — because that is the step nobody instruments.

The Four Failure Layers
  • 1
    Chunking — Fixed-size token splits violate semantic boundaries. The embedding model encodes fragments, not concepts. Retrieval degrades before a query is ever issued.
  • 2
    Retrieval quality — Bi-encoder cosine similarity measures topic proximity, not answer relevance. Without a reranker pass, you retrieve the right neighborhood but the wrong house.
  • 3
    Evaluation — End-to-end answer quality evals hide retrieval failures. A generator that improvises well can mask a recall@5 of 0.50 for weeks before it fails on the wrong query.
  • 4
    Infrastructure — Embedding drift, index staleness, and context window sizing are operational problems, not algorithmic ones. Teams solve them too late.

01 — The Setup

The Demo That Fooled Everyone

The demo works on 50 curated documents. The production system fails on 50,000 real ones. This is not a coincidence. It is a selection bias built into how most teams evaluate RAG before deploying it.

A typical proof-of-concept uses a small, clean corpus, asks questions whose answers exist verbatim in the documents, and tests with users who already know the right answer. The retrieval step returns the correct chunk because the corpus is small enough that almost anything retrieves correctly. The generator produces a good answer because the context it receives is always relevant.

Scale changes this. The same system with a 100x larger corpus, real user queries that do not phrase-match the documents, and content that only partially answers a question — that is where production happens. The demo did not test that scenario. The production system fails at it without warning.

I have seen this pattern at Infrrd on document intelligence pipelines three times in the last two years. Each time, the proof-of-concept benchmark looked strong. Each time, production performance degraded within weeks as query distribution shifted toward edge cases the demo corpus never contained.

02 — Chunking

Chunking Is Not a Strategy

Fixed-size chunking — every 512 tokens, every 1,024 tokens — is the default because it requires no decisions. That is also why it fails on anything more structured than a blog post.

The problem is semantic boundary violations. A 512-token chunk that starts mid-sentence, cuts a table in half, or separates a code block from its explanation passes structurally incoherent content to both the embedding model and the generator. The embedding captures a fragment, not a concept. The generator produces a hallucination or a non-answer because its context is incomplete.

On a 200-page insurance policy document chunked at fixed token boundaries, questions that span clause boundaries consistently miss. A definition in section 2 governs exclusions in section 14. Neither chunk retrieves on its own. Together they answer the question. Fixed chunking has no way to represent that relationship, so it never surfaces it.

The fix is document-aware segmentation: split on semantic boundaries — headings, paragraphs, list items, code blocks — not token counts. For structured documents, use the document's own structure as the split signal. For unstructured prose, sentence-boundary detection is the floor; paragraph-level segmentation is usually better.

One pattern that works well in practice: hierarchical chunking. Store chunks at two granularities — paragraph-level for retrieval, section-level for context expansion. Retrieve the paragraph, but pass the full parent section to the generator. This keeps retrieval precise while giving the generator enough context to reason across related content.

Python — document-aware chunking with parent context
from dataclasses import dataclass, field
from typing import List, Optional
import re

@dataclass
class Chunk:
    text:        str
    chunk_id:    str
    parent_id:   Optional[str]   # section this chunk belongs to
    parent_text: Optional[str]   # full section — passed to generator on retrieval hit
    metadata:    dict = field(default_factory=dict)

def split_by_headings(document: str, doc_id: str) -> List[Chunk]:
    # Split on markdown-style headings; adapt regex for your format
    section_pattern = re.compile(r'(?=^#{1,3}\s)', re.MULTILINE)
    sections = section_pattern.split(document)
    chunks = []

    for s_idx, section in enumerate(sections):
        section = section.strip()
        if not section:
            continue
        section_id = f"{doc_id}::s{s_idx}"

        # Paragraph-level sub-chunks within the section
        paragraphs = [p.strip() for p in section.split('\n\n') if p.strip()]
        for p_idx, para in enumerate(paragraphs):
            if len(para) < 40:   # skip headings-only stubs
                continue
            chunks.append(Chunk(
                text=para,
                chunk_id=f"{section_id}::p{p_idx}",
                parent_id=section_id,
                parent_text=section,   # full section for generator context
            ))
    return chunks

# On retrieval hit: index chunk.text, but pass chunk.parent_text to the LLM
Overlap is not the answer

Adding token overlap between chunks (e.g. 10% overlap) is a common workaround for boundary violations. It reduces the problem without solving it, and it inflates your index size by 10% with near-duplicate content. Use structural boundaries instead.

03 — Evaluation

The Retrieval Eval Gap

Most teams evaluate their RAG system at the output level: generate an answer, compare to a reference. This tells you whether the end-to-end pipeline is working. It does not tell you where it is failing.

The standard failure mode: a RAG eval shows 80% answer quality. Looks acceptable. But the retrieval step has a recall@5 of 0.52 — half the time, the correct document is not in the top 5 retrieved chunks. The generator is producing reasonable answers from the wrong context because it is a capable enough model to improvise. That improvisation masks the retrieval failure until you hit a query where the generator cannot fill the gap.

You need retrieval evals separate from generation evals. The metrics: recall@k (is the relevant document in the top k results?), precision@k (of the top k results, how many are relevant?), and mean reciprocal rank (how high is the first relevant result?). These require a labeled dataset — queries paired with their known-relevant documents. Building that dataset is the actual work most teams skip.

Without retrieval evals, you are navigating blind. An improvement to your chunking strategy might degrade recall@5 while improving the quality of hits when retrieval does work. You would see a flat or slightly improved end-to-end score and conclude the change was neutral. It was not.

Python — retrieval eval metrics
from typing import List, Dict

def recall_at_k(retrieved_ids: List[str], relevant_ids: List[str], k: int) -> float:
    """Fraction of relevant docs found in top-k retrieved."""
    top_k = set(retrieved_ids[:k])
    relevant = set(relevant_ids)
    if not relevant:
        return 0.0
    return len(top_k & relevant) / len(relevant)

def precision_at_k(retrieved_ids: List[str], relevant_ids: List[str], k: int) -> float:
    """Fraction of top-k that are relevant."""
    top_k = retrieved_ids[:k]
    relevant = set(relevant_ids)
    return sum(1 for doc_id in top_k if doc_id in relevant) / k

def mean_reciprocal_rank(retrieved_ids: List[str], relevant_ids: List[str]) -> float:
    """Position of the first relevant result (1-indexed)."""
    relevant = set(relevant_ids)
    for rank, doc_id in enumerate(retrieved_ids, 1):
        if doc_id in relevant:
            return 1.0 / rank
    return 0.0

def evaluate_retrieval(eval_set: List[Dict], retriever, k: int = 5) -> Dict:
    """
    eval_set: [{"query": str, "relevant_ids": [str, ...]}]
    retriever: callable(query) -> [chunk_id, ...]
    """
    recalls, precisions, mrrs = [], [], []
    for item in eval_set:
        retrieved = retriever(item["query"])
        recalls.append(recall_at_k(retrieved, item["relevant_ids"], k))
        precisions.append(precision_at_k(retrieved, item["relevant_ids"], k))
        mrrs.append(mean_reciprocal_rank(retrieved, item["relevant_ids"]))
    return {
        f"recall@{k}":    sum(recalls)    / len(recalls),
        f"precision@{k}": sum(precisions) / len(precisions),
        "mrr":            sum(mrrs)       / len(mrrs),
    }
How many labeled queries do you need?

For a retrieval eval to be statistically meaningful, I use 200 queries as a working minimum on a corpus of 10k–100k documents. Fewer than 100 and variance dominates — a single configuration change can swing your metrics by noise. Building the labeled set is mostly a one-time cost; maintaining it as the corpus evolves is the ongoing work.

04 — Context Assembly

Context Window Poisoning

Retrieving the wrong chunks is bad. Retrieving too many loosely-relevant chunks is a different kind of bad, and it is harder to detect because the generator still produces an answer.

The standard RAG pattern passes k retrieved chunks as context. Set k=10 and you have ten chunks that ranked highest by cosine similarity. On a specific query, that might mean two highly relevant chunks, four loosely related chunks, and four thematically adjacent chunks that do not contain the answer. The generator processes all ten and either dilutes the right signal with noise, or synthesizes across the irrelevant chunks and produces a confident wrong answer.

I call this context window poisoning. The model is not hallucinating from lack of information. It is hallucinating from excess noisy information. The generator treats all context as signal. When the context is 60% noise, it reasons on noise.

One concrete diagnostic: test your system with k=3 versus k=10. If k=3 outperforms k=10 on your query set, context window poisoning is active. The additional retrieved chunks are hurting. The fix is a reranker, covered in the next section, and a smaller final context window than you are probably using.

Symptom

Generator answers confidently but wrongly

Especially on queries where the right answer is in the corpus but adjacent documents discuss related concepts. The model blends them.

Fix

Rerank, then pass top 3–5 only

Retrieve k=20 with the bi-encoder, rerank to top 3–5 with a cross-encoder, pass only those to the generator. Short context, high precision.

05 — Infrastructure

Embedding Drift and Index Staleness

The embedding model and the vector index are two separate things that must stay in sync. Teams figure this out the hard way, usually six months after a migration.

Scenario: you build a RAG index using text-embedding-ada-002. You later migrate to text-embedding-3-large for better retrieval quality on new content. You use the new model for queries but the index still contains ada-002 embeddings. The cosine distances between query embeddings (new model space) and document embeddings (old model space) are now meaningless. Retrieval quality has silently collapsed. No error. No alert. Just bad answers.

The less dramatic version happens when you update your embedding model to a fine-tuned variant and re-embed only new documents. The index is now a hybrid of two embedding spaces. Queries retrieve inconsistently depending on which documents are relevant.

The operational rule: if you change the embedding model, rebuild the entire index. No partial migrations. This is expensive on large corpora, which is why teams skip it, which is why they pay in retrieval quality later.

Index staleness is a separate issue. Documents change. If your index is not updated when source documents are modified, retrieval returns stale content and the generator answers from outdated information. The fix is an update pipeline keyed on document modification time, not a periodic full re-index. Re-indexing only modified documents keeps the operational cost proportional to change volume, not corpus size.

Embedding model knowledge cutoffs

Embedding models have training cutoffs. If your corpus contains content that post-dates the embedding model's training data, vocabulary and concept representations may be poor for that content. Test this explicitly if you are indexing recent technical documentation — new framework names, API signatures, model identifiers. Observed symptom: retrieval quality degrades on recent content while performing well on older material.

06 — Retrieval Architecture

Why You Need a Reranker

A bi-encoder embedding model encodes query and document independently, then scores by cosine similarity. It is fast enough to search over millions of documents. It is also coarse: it captures topic similarity, not answer relevance.

A cross-encoder reranker processes the query and document together, producing a relevance score that accounts for their interaction. It cannot precompute document representations, so it runs 10–100x slower than the bi-encoder. But it answers a different question: not "does this document discuss the same topic as the query" but "does this document actually answer the query."

The standard production pattern: retrieve k=20 candidates with the bi-encoder, rerank with a cross-encoder, pass top 3–5 to the generator. Retrieval is fast and high-recall. Reranking is slower but high-precision. The generator gets a short, relevant context window.

On internal benchmarks with a 50k-document insurance corpus: bi-encoder alone produced recall@3 of 0.61. Cross-encoder reranking on top-20 candidates produced recall@3 of 0.84. Same underlying index, same bi-encoder retrieval step. The reranker added 23 percentage points of recall with no change to indexing.

Python — bi-encoder retrieve + cross-encoder rerank pipeline
from sentence_transformers import CrossEncoder
from typing import List

# Load once at startup — not per request
reranker = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")

def retrieve_and_rerank(
    query: str,
    vector_store,           # your Pinecone / Weaviate / pgvector client
    initial_k: int = 20,
    final_k:   int = 5,
) -> List[str]:
    # Stage 1: bi-encoder retrieval — fast, high-recall
    candidates = vector_store.query(query, top_k=initial_k)
    # candidates: [{"id": str, "text": str, "score": float}]

    if not candidates:
        return []

    # Stage 2: cross-encoder reranking — slower, high-precision
    pairs = [(query, c["text"]) for c in candidates]
    scores = reranker.predict(pairs)

    ranked = sorted(
        zip(candidates, scores),
        key=lambda x: x[1],
        reverse=True
    )

    # Return text of top final_k — this is what goes to the generator
    return [c["text"] for c, _ in ranked[:final_k]]
Do not skip the reranker to save latency

The reranker adds 100–300ms depending on model size and k. The recall regression from removing it is larger than the UX improvement from saving 300ms. Optimize the bi-encoder retrieval and generation streaming steps first. The reranker is the last thing to trade away.

07 — Performance

The Latency Wall

A production RAG pipeline at scale has a latency profile most teams do not model before they build it.

Bi-encoder retrieval from a well-optimized vector store (Pinecone, Weaviate, pgvector on a tuned instance): 20–80ms for k=20 over 1M documents. Cross-encoder reranking of 20 candidates: 100–300ms. Generator call (Claude Sonnet 4, streaming): 800–2,000ms to first token. Total before the user sees anything: 1–2.5 seconds.

That is acceptable for many applications and not for others. The places where it breaks: conversational interfaces that expect sub-second first-token latency, high-volume APIs where generator call queuing creates backpressure, and real-time search overlays.

  • Stream generation immediately. Do not wait for the full generator response before sending output. Streaming passes first-token latency perception to the user. It is free and every production RAG system should do it.
  • Cache retrieval results for common queries. A query cache keyed on a normalized query hash, with a TTL of 5–15 minutes, eliminates retrieval and reranking latency for repeated queries. Effective for FAQ-style systems and search overlays. Less effective for conversational pipelines with high query diversity.
  • Retrieval caching at the embedding level. If the same document is retrieved across many queries, the reranker inference is redundant. Cache (query_embedding, doc_id) to rerank score at the pair level. Useful for corpora where a small set of documents answers the majority of queries.
  • Async pre-retrieval where possible. In conversational flows, pre-fetch likely relevant context based on early signals — topic, previous turns, detected intent — before the user submits the final query. The retrieval step can complete in the background while the user is still typing.
08 — Applicability

When RAG Actually Works

RAG has a specific failure profile. It also has a specific success profile. Both are worth being precise about.

Condition RAG fits Why
Corpus too large to fine-tune or changes frequently Yes RAG updates with the corpus; fine-tuning requires retraining cycles
Answers exist verbatim or near-verbatim in source documents Yes Retrieval finds them; generator extracts, not synthesizes
Queries are specific, factual, not reasoning-heavy Yes Bi-encoder + reranker handles factual proximity well
Corpus fits in a long-context window (under ~200k tokens) No Full-context retrieval is simpler, cheaper, and more accurate
Task requires synthesis across many documents simultaneously No Retrieval selects a subset; synthesis requires the full picture
Multi-step reasoning where each step informs the next query No Agentic retrieval loops are needed; standard RAG is single-shot

The specific case I see misused most often: internal knowledge base Q&A on a 200-document corpus. Teams build a full RAG pipeline that would be more simply and more accurately handled by passing all 200 documents in a 200k-token context window and asking the question once. The RAG pipeline introduces retrieval risk on a dataset small enough to avoid it entirely. The complexity is not justified by the corpus size.

The threshold where RAG starts to pay off over full-context is roughly 500–1,000 documents for most embedding model and context window combinations, depending on average document length. Below that, reach for full context first and add retrieval only when you hit actual latency or cost constraints.

09 — Takeaway

The Frame to Leave On

Most RAG failures happen at the seams. Between the chunking strategy and the embedding model. Between the retrieval step and the generation step. Between the evaluation methodology and the actual failure modes. The components test fine individually. The system fails in production because the integration assumptions were wrong.

The sequence that works: build retrieval evals before you build a reranker. Build a reranker before you tune the generator. Diagnose where the pipeline is failing before you optimize anything. The failure is almost always in the retrieval layer, but teams optimize the generation layer because that is where the output is visible. You are measuring the wrong thing and fixing the wrong step.

The architecture is sound. The defaults ship broken.
