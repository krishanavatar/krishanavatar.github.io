Production Engineering — LLM Cost

LLM Cost Optimization in Production

How I reduced inference spend by 60% on an enterprise IDP platform using prompt caching, model routing, and token-level prompt engineering — without touching accuracy.

krishanavatar.github.io / July 2026 / 10 min read
CONTEXT

This is drawn from experience building a production document intelligence system — high-volume, structured-document extraction, multi-provider. The framework is provider-agnostic. Numbers are from a live benchmark set unless labeled anecdotal.

The Three Levers
  • 1
    Caching — Prompt caching at the provider level eliminates redundant inference on repeated system prompts and static context. On document pipelines, this alone cuts 40–50% of token spend.
  • 2
    Model routing — Not every extraction task needs GPT-4o or Claude Opus. Routing simple classification to a smaller model and reserving the frontier model for ambiguous cases reduces cost by 30–40% with no accuracy regression on clear-cut documents.
  • 3
    Prompt compression — Verbose prompts are a tax. Structured, minimal prompts with explicit output schemas reduce input tokens by 25–35% and improve parse reliability.

01 — The Problem

The Invoice That Triggered This

At some point in a production LLM system, the billing dashboard stops being background noise and starts being a line item in a business case. For us, that moment came when a single month's inference bill crossed a threshold that made the product commercially unviable at scale. The accuracy was good. The throughput was acceptable. The cost was not.

The documents were structured forms — high-volume, repetitive layouts, same fields in roughly the same positions every time. A major insurance carrier we were building for processed tens of thousands of these monthly. Every page goes through OCR, then LLM extraction, then validation. The LLM step was the one eating money. And the irony was that most of the spend wasn't on hard documents. It was on easy ones — the standard forms that arrived clean and consistent, getting sent to the same frontier model regardless.

That's not a model problem. That's a routing problem. And once you see it that way, the fix is mechanical.

02 — Architecture

Platform Context

The system routes documents through a pipeline: ingest → OCR → classification → LLM extraction → validation → output. The LLM extraction step is where we call the model to produce structured JSON from raw OCR text, guided by a schema and a system prompt.

Three providers run in production: OpenAI (GPT-4o, GPT-4o-mini), Google (Gemini 2.5 Flash, Gemini 2.5 Pro — both via Vertex AI), and Anthropic (Claude Haiku 4.5, Claude Sonnet 4). The provider and model tier are config-driven — no code changes required to switch. That config layer is what makes the optimization tractable. Without it, cost reduction is a manual migration each time.

llm-config.yaml — provider config
# Provider config — drives model selection per document type and confidence tier
llm:
  default_provider: openai
  providers:
    openai:
      high_confidence:  gpt-4o-mini     # cheap, fast
      low_confidence:   gpt-4o          # frontier, ambiguous docs
    anthropic:
      high_confidence:  claude-haiku-4-5-20251001
      low_confidence:   claude-sonnet-4-20250514
    google:
      high_confidence:  gemini-2.5-flash  # via Vertex AI
      low_confidence:   gemini-2.5-pro    # via Vertex AI
  routing:
    confidence_threshold: 0.85   # OCR confidence from prior step
    fallback_to_frontier: true
03 — Caching

Prompt Caching: The Biggest Win

Prompt caching is the single highest-ROI optimization for document pipelines. The idea is simple: if the same system prompt and schema appear in every request, the provider can cache the KV computation from those tokens and charge you a fraction of the input price for subsequent hits. Anthropic charges 10% of the normal input rate for cache hits. OpenAI charges 50%.

On our platform, the system prompt and schema together run about 800–1,200 tokens per request. With 50,000 documents a month, the uncached cost of just that static context is non-trivial. With caching enabled and cache hit rates above 90%, those tokens cost 10 cents on the dollar.

The implementation requirement: the cached prefix must be structurally stable. Any variation in the system prompt — even a timestamp injected at runtime — breaks the cache. We audited our prompts and removed every dynamic element from the cacheable prefix, pushing variable content (the document text itself) to the user turn only.

Python — Anthropic prompt caching
import anthropic

client = anthropic.Anthropic()

# System prompt and schema are static — mark for caching
# Cache TTL: 5 minutes (Anthropic). Reset on any content change.
response = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": SYSTEM_PROMPT,         # static, ~600 tokens
            "cache_control": {"type": "ephemeral"}
        },
        {
            "type": "text",
            "text": OUTPUT_SCHEMA,        # static, ~400 tokens
            "cache_control": {"type": "ephemeral"}
        }
    ],
    messages=[
        {
            "role": "user",
            "content": f"Extract from this document:\n\n{ocr_text}"
            # dynamic content goes here — NOT in the cache prefix
        }
    ]
)

# Check cache performance in response headers
usage = response.usage
cache_hit  = usage.cache_read_input_tokens
cache_miss = usage.cache_creation_input_tokens
print(f"Cache hit: {cache_hit} tokens @ 10% price")
print(f"Cache miss: {cache_miss} tokens @ 125% price (write overhead)")
Cache write overhead

The first call that populates the cache costs 125% of the normal input price (Anthropic) or 100% (OpenAI, no write surcharge). On a high-volume pipeline, this is noise — the write overhead is recovered within 3–5 subsequent hits. On low-volume or sporadic workloads, caching is not worth it.

One more thing worth knowing: Anthropic's cache TTL is 5 minutes. If your pipeline has periods of inactivity longer than that, cached tokens expire and the next request pays the write price again. We added a lightweight keepalive — a minimal ping request every 4 minutes during idle periods — to hold the cache warm during business-hours processing windows.

Vertex AI Context Caching — Gemini 2.5

Vertex AI ships its own context caching layer for Gemini 2.5 Flash and Pro. The mechanics differ from Anthropic's: instead of marking a prefix in the API call, you create a CachedContent object ahead of time and attach it to subsequent model calls. The TTL is configurable up to 1 hour (versus Anthropic's fixed 5 minutes), which makes it better suited for pipelines with longer inter-batch gaps.

The tradeoff: Vertex AI's context caching has a minimum cached token threshold — your static context must be large enough to qualify. For small system prompts this is a non-starter, but for pipelines that load large schemas, few-shot extraction examples, or document templates, it's the right lever. Gemini 2.5 Flash also supports thinking budget control separately, so you can suppress chain-of-thought on the cheap tier and pay only for output tokens.

Python — Vertex AI context caching with Gemini 2.5 Flash
import vertexai
import datetime
from vertexai import caching
from vertexai.generative_models import GenerativeModel, Content, Part

vertexai.init(project="my-gcp-project", location="us-central1")

# Create a CachedContent object for the static prefix.
# TTL configurable to 1 hour — outlasts Anthropic's 5-min window.
cached_content = caching.CachedContent.create(
    model_name="gemini-2.5-flash",
    display_name="doc-extraction-context",
    contents=[
        Content(role="user", parts=[
            Part.from_text(SYSTEM_PROMPT + "\n\n" + OUTPUT_SCHEMA)
            # large static context: prompt + schema + few-shot examples
        ])
    ],
    ttl=datetime.timedelta(hours=1),
)

# Attach cached content to the model — all requests share the KV cache
model = GenerativeModel.from_cached_content(cached_content)

response = model.generate_content(
    f"Extract from this document:\n\n{ocr_text}",
    generation_config={
        "response_mime_type": "application/json",
        "thinking_config": {"thinking_budget": 0}  # disable thinking on cheap tier
    }
)

# Cache usage in response metadata
meta = response.usage_metadata
print(f"Cached: {meta.cached_content_token_count} tokens")
print(f"Prompt: {meta.prompt_token_count} tokens")
print(f"Output: {meta.candidates_token_count} tokens")
Vertex vs Direct API

Vertex AI adds IAM, VPC-SC, and audit logging that the direct Google AI API lacks — relevant for enterprise compliance. The model IDs are the same (gemini-2.5-flash, gemini-2.5-pro). If you're already on GCP, Vertex is the right path; otherwise the direct SDK is simpler to start with but loses the longer cache TTL.

04 — Model Selection

Model Routing: Right Tool, Right Task

Sending every document to a frontier model is the same mistake as using a sledgehammer to hang a picture frame. The frontier models (GPT-4o, Claude Sonnet 4, Gemini 2.5 Pro) are priced for their capability on hard problems. Standard structured forms with clean OCR are not hard problems.

Our routing logic uses OCR confidence as the primary signal. If the confidence score from the OCR step is above 0.85 — meaning the text was extracted cleanly, no smudges, no handwriting, no degraded scans — the document goes to the cheaper model tier. Below 0.85, it escalates to the frontier model. We tuned this threshold over four weeks of production data, comparing extraction accuracy across tiers at different confidence levels.

Model Provider Input $/1M Output $/1M Use case
GPT-4o-mini OpenAI $0.15 $0.60 High-confidence docs
GPT-4o OpenAI $2.50 $10.00 Ambiguous, complex
Claude Haiku 4.5 Anthropic $0.80 $4.00 High-confidence docs
Claude Sonnet 4 Anthropic $3.00 $15.00 Ambiguous, complex
Gemini 2.5 Flash Google / Vertex $0.15 $0.60 High-confidence docs
Gemini 2.5 Pro Google / Vertex $1.25 $10.00 Ambiguous, complex

In practice, about 72% of our document volume clears the 0.85 confidence threshold. That 72% goes to the cheaper tier. The 28% that fails — degraded scans, handwritten fields, multi-column layouts — goes to the frontier model. The blended per-document cost dropped from $0.018 to $0.007, anecdotally. The extraction accuracy on the high-confidence tier was within 0.3% of the frontier model on our benchmark set.

One pattern worth building: a shadow mode. Before committing to the cheaper model in production, run both models on the same documents for two weeks and compare outputs. I've seen cases where a model that looks cheaper on paper performs badly on a specific document format and forces expensive retries that wipe out the savings.

Python — OCR-confidence-based model router
from dataclasses import dataclass
from enum import Enum

class Tier(Enum):
    CHEAP    = "cheap"
    FRONTIER = "frontier"

MODEL_MAP = {
    "openai":    {Tier.CHEAP: "gpt-4o-mini",                Tier.FRONTIER: "gpt-4o"},
    "anthropic": {Tier.CHEAP: "claude-haiku-4-5-20251001",  Tier.FRONTIER: "claude-sonnet-4-20250514"},
    "google":    {Tier.CHEAP: "gemini-2.5-flash",           Tier.FRONTIER: "gemini-2.5-pro"},
}

def select_model(provider: str, ocr_confidence: float, threshold: float = 0.85) -> str:
    tier = Tier.CHEAP if ocr_confidence >= threshold else Tier.FRONTIER
    model = MODEL_MAP[provider][tier]
    return model

# Usage
model = select_model("anthropic", ocr_confidence=0.91)
# → "claude-haiku-4-5-20251001"

model = select_model("anthropic", ocr_confidence=0.62)
# → "claude-sonnet-4-20250514"
05 — Prompt Engineering

Prompt Compression: Paying for What You Need

Most LLM prompts I've seen in production are verbose in ways that add no extraction value. They're written the way people write requirements documents — with hedges, explanations, and repetition. Every one of those tokens costs money and, past a point, degrades output quality by diluting the instruction signal.

I audited our extraction prompts when this cost work started. The original system prompt was 1,847 tokens. After removing explanation paragraphs that described what IDP is (the model knows), removing redundant field descriptions already covered by the schema, and restructuring the output instructions to be command-form rather than narrative, the prompt dropped to 612 tokens. Extraction accuracy on our benchmark: identical.

Before

Narrative prompt

1,847 tokens. Explains context the model doesn't need. Redundant field descriptions. Output format described in paragraph form.

After

Structured prompt

612 tokens. Command-form instructions only. Fields defined once in the schema. Output format specified as a JSON example, not prose.

Prompt — before vs after
# BEFORE: verbose, 1847 tokens
"""
You are an intelligent document processing assistant specialized in
extracting structured information from insurance documents. Your role
is to carefully analyze the provided OCR text from insurance forms
and extract relevant information according to the schema provided.

When extracting information, please ensure you:
1. Read the entire document carefully before making any extractions
2. Pay attention to field labels and their corresponding values
3. Handle missing fields by returning null values
4. Ensure all dates are formatted as YYYY-MM-DD
...
(continues for 40 more lines)
"""

# AFTER: command-form, 612 tokens
"""
Extract structured data from insurance OCR text per the schema below.

Rules:
- Missing fields → null (never omit)
- Dates → YYYY-MM-DD
- Amounts → numeric, no currency symbols
- Confidence < 0.7 on any field → set field_confidence: "low"

Output: valid JSON matching the schema exactly. No prose.
"""

Two specific patterns worth knowing. First: move your output schema to a separate cache block rather than embedding it inline in the system prompt. The schema changes rarely; the instructions might change more often. Splitting them into two cacheable blocks means a schema update doesn't invalidate your instruction cache. Second: use JSON mode or structured outputs wherever the provider supports it. The output becomes more deterministic, and you eliminate the token overhead of the model wrapping its JSON in markdown code fences.

Python — OpenAI structured outputs
from pydantic import BaseModel
from typing import Optional
import openai

class InsuranceExtraction(BaseModel):
    policy_number:   str
    insured_name:    str
    effective_date:  str             # YYYY-MM-DD
    premium_amount:  float
    coverage_type:   str
    claim_number:    Optional[str]   # null if not a claim form

client = openai.OpenAI()

response = client.beta.chat.completions.parse(
    model="gpt-4o-mini",
    messages=[
        {"role": "system", "content": SYSTEM_PROMPT},
        {"role": "user",   "content": ocr_text}
    ],
    response_format=InsuranceExtraction,  # no JSON parsing, no markdown fences
)

extraction = response.choices[0].message.parsed
# → InsuranceExtraction object, validated by Pydantic
06 — The Config Layer

The Config Layer That Makes It Maintainable

Three separate optimization levers — caching, routing, prompt compression — become a maintenance problem if each one is hardcoded. A config layer is what makes the whole system observable and adjustable without deploys.

The LLM orchestrator reads provider config at startup, selects the model based on the routing rules, builds the cacheable prompt blocks, and emits cost metrics to Grafana. When we wanted to benchmark Gemini 2.5 Flash versus GPT-4o-mini on the cheap tier, the comparison was a config change and a 24-hour traffic split — no code change, no deployment.

Python — LLM orchestrator with cost tracking
import time
from dataclasses import dataclass, field
from typing import Dict, Any

@dataclass
class ExtractionResult:
    data:              Dict[str, Any]
    model_used:        str
    provider:          str
    input_tokens:      int
    output_tokens:     int
    cache_hit_tokens:  int
    latency_ms:        float
    estimated_cost:    float           # USD

class LLMOrchestrator:
    COST_PER_M = {
        "gpt-4o-mini":               {"in": 0.15,  "out": 0.60},
        "gpt-4o":                     {"in": 2.50,  "out": 10.00},
        "claude-haiku-4-5-20251001":  {"in": 0.80,  "out": 4.00},
        "claude-sonnet-4-20250514":   {"in": 3.00,  "out": 15.00},
        "gemini-2.5-flash":           {"in": 0.15,  "out": 0.60},
        "gemini-2.5-pro":             {"in": 1.25,  "out": 10.00},
    }

    def __init__(self, config: dict):
        self.config   = config
        self.provider = config["llm"]["default_provider"]

    def extract(self, ocr_text: str, ocr_confidence: float) -> ExtractionResult:
        model = select_model(self.provider, ocr_confidence,
                             self.config["llm"]["routing"]["confidence_threshold"])
        t0       = time.time()
        response = self._call(model, ocr_text)       # provider-specific call
        latency  = (time.time() - t0) * 1000

        cost = self._estimate_cost(model, response.usage)
        self._emit_metrics(model, cost, latency)     # → Grafana

        return ExtractionResult(
            data=response.parsed,
            model_used=model,
            provider=self.provider,
            input_tokens=response.usage.input_tokens,
            output_tokens=response.usage.output_tokens,
            cache_hit_tokens=getattr(response.usage, "cache_read_input_tokens", 0),
            latency_ms=latency,
            estimated_cost=cost,
        )

    def _estimate_cost(self, model: str, usage) -> float:
        rates = self.COST_PER_M.get(model, {"in": 0, "out": 0})
        cache_hit   = getattr(usage, "cache_read_input_tokens", 0)
        normal_in   = usage.input_tokens - cache_hit
        return (normal_in * rates["in"] + cache_hit * rates["in"] * 0.1
                + usage.output_tokens * rates["out"]) / 1_000_000
07 — Results

What the Numbers Looked Like

After running all three optimizations in production for 60 days, here's what changed. The numbers are internal and the baseline is the per-document blended cost before the work started.

Optimization Before After Reduction
Prompt caching (system + schema) $0.018/doc $0.011/doc ~39%
Model routing (cheap/frontier split) $0.011/doc $0.0075/doc ~32%
Prompt compression (1847 → 612 tokens) $0.0075/doc $0.0064/doc ~15%
Combined (50k docs/month) $900/mo $320/mo ~64%

Extraction accuracy on the production benchmark set (1,200 manually labeled documents): before the work, 97.2%. After: 96.9%. The 0.3% regression is within noise on that sample size and attributable to the cheap-tier model on edge cases at the 0.85 threshold boundary — not to the prompt compression or caching changes. Raising the threshold to 0.87 recovered it entirely at the cost of routing 4% more volume to the frontier tier.

08 — Failure Modes

Where This Breaks

None of this is universal. Three scenarios where I've seen these optimizations either fail or not apply.

  • Low-volume or bursty pipelines. Prompt caching needs sustained throughput to amortize the cache write cost. If your pipeline processes 200 documents a day spread unevenly, you're paying the 125% write price more often than you're hitting the 10% cache price. Below roughly 2,000 requests per hour for a given prompt, caching is often a wash.
  • Highly variable document types. The 0.85 OCR confidence threshold works for a specific document corpus. A corpus with significant format variation — handwritten claims, multi-language forms, photographs of paper documents — will have a different accuracy curve at that threshold. The routing split needs to be calibrated per corpus, not set once.
  • Tasks where output length dominates cost. All three of these optimizations reduce input token cost. If your use case produces long outputs (summarization, report generation, free-form analysis), output tokens will dominate your bill and none of this moves the number meaningfully. For output-heavy workloads, the main levers are model selection and output length constraints, not prompt caching.
On accuracy regressions

Run a shadow mode before committing to the cheaper tier in production. Two weeks of parallel inference on a sampled subset, comparing outputs, before you flip the routing switch. The cost is negligible compared to a production accuracy regression on a live enterprise client.

09 — Takeaway

The Frame to Leave On

LLM cost is not a fixed property of a use case. It's an artifact of how you've wired the request. Most production systems I've seen send the same model, the same verbose prompt, and the same static context on every call, regardless of document difficulty. That's a routing and caching problem, not a model problem.

The three levers — caching, routing, compression — compound. Each one assumes a baseline that the previous one improved. Apply them in order: caching first (biggest bang, no accuracy risk), routing second (significant savings, requires calibration), compression last (smaller gains, but permanently reduces your cost floor). Don't compress prompts before you've confirmed caching is working; the interaction between cache invalidation and prompt changes is a source of subtle bugs.

The config layer is not optional. Without it, every optimization is a one-time migration. With it, cost optimization is a runtime knob.
